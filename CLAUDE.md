# CLAUDE.md


## 1. 编码前先思考
**不做臆测，不掩饰困惑，主动点明权衡取舍。**

实现需求前：
- 明确说明你的前提假设。若存在不确定，主动询问。
- 若存在多种解读方式，全部列出——不要私自选定其一。
- 若存在更简单的实现方案，主动告知。必要时提出异议。
- 若有内容表述不清，立刻停下。指出困惑点，向用户确认。

## 2. 简洁优先
**用最少的代码解决问题，绝不添加推测性功能。**

- 不实现超出需求范围的功能。
- 不为仅单次使用的代码做抽象封装。
- 不增加未被要求的“灵活性”或“可配置性”。
- 不对不可能出现的场景做异常处理。
- 如果你写了200行代码但实际50行就能解决，立刻重写。

扪心自问：“资深工程师会不会觉得这设计过度复杂？” 如果答案是肯定的，就做简化。

## 3. 外科手术式精准改动
**只修改必要内容，仅清理自己改动产生的冗余。**

修改已有代码时：
- 不擅自“优化”相邻的代码、注释或格式。
- 不对运行正常的代码做重构。
- 遵循现有代码风格，即便你有更优的写法。
- 若发现无关的废弃代码，可以提及，但不要直接删除。

因你的改动产生冗余代码时：
- 仅移除因本次改动而不再使用的导入、变量或函数。
- 除非用户要求，否则不删除原本就存在的废弃代码。

检验标准：每一行被修改的代码，都应能直接对应到用户的需求。

## 4. 目标驱动执行
**先定义成功标准，循环验证直到达成。**

将任务转化为可验证的目标：
- “新增校验” → “为非法输入编写测试用例，再让测试全部通过”
- “修复Bug” → “编写可复现问题的测试用例，再让测试通过”
- “重构X模块” → “确保重构前后测试均能正常通过”

对于多步骤任务，先给出简要计划：
```
1. [步骤内容] → 验证项：[检查点]
2. [步骤内容] → 验证项：[检查点]
3. [步骤内容] → 验证项：[检查点]
```
## 5. 不要让大模型执行非语言类工作
**重试策略、路由逻辑、告警升级阈值都应落在确定性代码中。**
## 6. Token 预算为硬性约束，无任何例外
**若任务有超出预算的趋势，立即停止并向用户确认。**
## 7. 暴露冲突，不做折中调和
**若代码库中两处逻辑存在分歧，明确指出矛盾并询问以哪一处为准。**
## 8. 先读再写。
**新增代码前，先理解周边代码逻辑（当前文件及同目录关联文件）。**
## 9. 测试是必要手段，而非最终目标
**能通过但未校验有效业务逻辑的测试毫无价值，测试必须验证实际行为。**
## 10.  长耗时操作必须设置检查点
**每完成一个关键步骤，总结已完成内容并确认后再推进下一步。**
## 11.  约定优于创新
**在成熟代码库中，即便存在“更优”方案，也应遵循现有编码模式。**
## 12.  错误显性化，禁止静默失败
**逐条披露所有跳过的记录、回滚的事务、违反的约束；但凡有逻辑被绕过，绝不能上报为执行成功。**

## Agent skills

### Issue tracker

Issues 作为本地 markdown 文件存放在 `.scratch/<feature-slug>/` 下。See `docs/agents/issue-tracker.md`.

### Triage labels

五个标签均使用默认名：`needs-triage`, `needs-info`, `ready-for-agent`, `ready-for-human`, `wontfix`。See `docs/agents/triage-labels.md`.

### Domain docs

Single-context layout: 根目录 `CONTEXT.md` + `docs/adr/`。See `docs/agents/domain.md`.
